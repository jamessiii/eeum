# Group > Card 패턴

## 목적

그룹 목록과 그룹 안의 카드 목록을 함께 관리해야 하는 화면을 위한 재사용형 UI 패턴.

다음 조건에서 사용한다.

- 화면에 `그룹 -> 아이템` 관계가 분명해야 할 때
- 사용자가 빠르게 훑고, 추가하고, 수정하고, 정렬하고, 숨기고, 삭제해야 할 때
- 표보다는 카드가 대상의 정체성을 더 잘 보여줄 때

다음 조건에서는 쓰지 않는다.

- 화면이 거의 스프레드시트처럼 동작할 때
- 한 화면에 너무 많은 항목을 높은 밀도로 보여줘야 할 때
- 계층이 2단계를 넘어갈 때

## 기본 구조

패턴은 다섯 레이어로 구성한다.

1. 보드 셸
2. 그룹 섹션
3. 카드 그리드
4. 추가 카드
5. 숨김 패널

### 1. 보드 셸

- 페이지 레벨의 큰 컨테이너 하나
- 제목, 설명, 소규모 유틸리티 액션
- 아래에는 그룹 섹션이 세로로 쌓임

### 2. 그룹 섹션

- 각 그룹은 독립된 섹션 카드
- 섹션 헤더에는 다음이 들어감
  - 그룹 제목
  - 메타 정보 또는 개수
  - 수정 아이콘 같은 작은 액션
- 섹션 바디에는 하나 이상의 카드 그리드가 들어감

### 3. 카드 그리드

- 카드는 가로 방향 그리드 안에 배치
- 같은 그리드 안에서는 일정한 크기와 리듬 유지
- 추가 카드는 일반 카드와 같은 크기여야 함

### 4. 추가 카드

- 그리드의 마지막 또는 보드의 마지막에 위치
- 특정 그룹에 귀속되는 항목이라면 상단 우측 큰 버튼보다 추가 카드를 우선
- 레이아웃 밖의 CTA가 아니라 레이아웃 내부의 일부처럼 보여야 함

### 5. 숨김 패널

- 숨긴 항목은 완전히 사라지지 않아야 함
- 별도 패널에서 복원 가능해야 함
- 숨김 패널 안에서도 원래 카드의 정체성이 어느 정도 유지돼야 함

## 배치 규칙

### 그룹 배치

- 그룹은 세로로 쌓는다
- 각 그룹은 하나의 독립 단위처럼 보여야 한다
- 그룹 간 간격은 카드 간 간격보다 더 넓어야 한다
- 그룹 하나를 볼 때 내부 구조가 한눈에 읽혀야 한다

### 카드 배치

- 같은 그리드 안에서는 카드 폭을 일정하게 유지
- 가능하면 높이도 일정하게 유지
- 좌상단: 핵심 정체성
- 우상단: 수정 액션
- 중간: 보조 메타데이터
- 우하단: 상태 배지 또는 분류 배지

### 카드 내부 정보 위계

권장 순서:

1. 작은 kicker 또는 타입 라벨
2. 메인 타이틀
3. 보조 정보 1~2줄
4. 상태 배지 또는 분류 pill

원칙:

- 타이틀이 항상 우선
- 메타 정보는 식별을 돕는 수준까지만
- 상태 배지는 짧고 조용하게
- 관리 화면에서는 대시보드형 수치를 카드에 과하게 넣지 않기

## 추가 패턴

권장 흐름:

1. 추가 카드 클릭
2. 새 그룹 또는 새 카드 즉시 생성
3. 새 항목 제목을 인라인 수정 상태로 전환
4. 상세 필드는 필요할 때 모달 수정

이유:

- 생성 전에 폼을 먼저 띄우는 것보다 빠름
- 레이아웃 흐름이 끊기지 않음
- 사용자가 생성 위치를 더 잘 이해함

원칙:

- 그룹 생성은 보드 하단
- 카드 생성은 대상 그리드의 마지막
- 빈 상태에서도 같은 추가 카드 언어를 유지

## 수정 패턴

수정은 두 단계로 나눈다.

- 인라인 수정
  - 그룹 제목, 짧은 카드 제목
- 모달 수정
  - 구조화된 필드, 관계, 옵션, 메모, 타입, 연결 정보

원칙:

- 수정 affordance는 화면마다 통일
- 수정 버튼은 작은 아이콘형 우선
- 사용자가 이미 대상을 알고 있는 경우 인라인 우선
- 여러 필드나 관계를 만지는 경우 모달 우선

## 드래그 패턴

### 정렬

- 드래그되는 대상은 원본과 같은 모양을 유지
- 커스텀 고스트는 원본 카드 또는 원본 섹션을 그대로 닮아야 함
- 고스트는 포인터를 끊김 없이 따라와야 함
- 드롭 위치는 거리 기반으로 자연스럽게 결정

### 액션 드롭존

사이드 액션 존으로 처리할 수 있는 것:

- 숨기기
- 삭제
- 보관

원칙:

- 드래그 중 전역적으로 보이게
- 화면마다 같은 위치 사용
- 문구는 짧게
- 실제 타깃에 들어왔을 때만 강한 반응

### 그룹 간 이동

이 규칙은 화면마다 명시해야 한다.

- 그룹 간 이동 허용
- 같은 그룹 안에서만 허용
- 다른 카드에 드롭하면 연결 액션으로 변환

이 부분은 애매하게 두지 않는다.

### 드래그 피드백

드래그 중에는 다음이 분리되어야 한다.

- 원본 카드의 드래그 상태
- 마우스를 따라오는 고스트
- 드롭 타깃 강조
- 액션 드롭존 활성화

피해야 할 것:

- 동시에 여러 움직이는 피드백
- 드래그 중 원본과 전혀 다른 모양으로 바뀌는 것
- hover 효과가 drag 상태보다 더 강한 것

## 빈 상태 패턴

빈 상태에서도 같은 구조 언어를 유지한다.

권장 방식:

- 보드 셸은 그대로 유지
- 섹션 또는 추가 카드 문법도 그대로 유지
- 첫 추가 또는 첫 업로드 이후 무엇이 생기는지만 설명

완전히 다른 빈 상태 UI로 바꾸지 않는다.

## 인터랙션 톤

- hover: 약하게
- active target: 분명하게
- edit affordance: 작고 일관되게
- add affordance: 눈에 띄지만 레이아웃 안쪽에서
- destructive action: 즉시 실행보다 드롭 또는 확인을 거치게

이 패턴은 홍보성보다 운영성에 가까워야 한다.

## 재사용 가능한 결정

다른 앱에서도 재사용 가능한 결정:

- 밀도보다 정체성이 중요하면 테이블 대신 `그룹 -> 카드`
- 위치가 중요한 생성은 상단 버튼보다 하단 추가 카드
- 제목 수정은 인라인, 상세 수정은 모달
- 숨기기/삭제는 사이드 드롭존
- 카드 안 정보는 정체성, 메타, 상태 배지 중심

## 구현 체크리스트

- 보드 셸 존재
- 그룹 섹션 세로 스택
- 카드 그리드 리듬 일관
- 추가 카드 크기 정합
- 제목 위계 분명
- 수정 affordance 일관
- 인라인 생성 흐름 지원
- 드래그 고스트가 원본과 같은 형태
- 드롭 규칙 명시
- 숨김/삭제 복구 가능 여부 정리
- 빈 상태에서도 같은 레이아웃 언어 유지

## 한 줄 요약

`세로 그룹과 컴팩트 카드로 구성된 보드에서, 인라인 생성과 모달 상세 수정, 그리고 원본 형태를 유지하는 드래그 액션을 결합한 패턴.`
